# 项目 TODO：当前阶段的不足与潜在问题

> 截止：2026-07-14，temperature=0.8，共 8 个工具，分类记忆已上线，用户输入自动记录已上线。
> 优先级：P0 = 严重影响体验，应尽快处理；P1 = 架构层面，做了就上台阶；P2 = 体验打磨；P3 = 技术债务/未来隐患。

---

## 🔴 P0：严重影响体验，应尽快处理

### 1. 单次任务模式 vs 持续对话模式

**问题**：当前 `run_agent(task)` 是"输入一个问题 → 模型回答 → 程序结束"的单次任务模式。用户想继续聊"其实是因为..."——得重新运行脚本，完全没有上下文延续感。

**影响**：这是"工具助手"和"陪伴助手"的本质区别。陪伴感要求"你一句、我一句，聊下去"，而不是"答完一道题就跑"。

**解决方向**：
- 让 `run_agent` 内部变成 while True 循环，用户输入 `exit`/`再见` 才退出
- 每一轮对话都追加到 messages，保持上下文
- 预计改动：`agent.py` ~50 行，`main.py` ~10 行

### 2. 对话历史不持久化

**问题**：每次调用 `run_agent`，`messages = [system, user]` 都是全新的列表。上次对话的"你说过什么、助手怎么回答的"完全不在上下文中。

**影响**：就算记忆库里存了"用户今天被老板批评了"，助手下次启动时也只是看到一条独立的记录，不知道"我们上次聊到哪了"。

**解决方向**：
- 对话历史存到 `data/conversation.json`（或 `data/messages.json`）
- 启动时从文件加载，继续之前的对话
- 预计改动：`memory.py` ~30 行，`agent.py` ~15 行

### 3. 用户输入自动存了，但存得不够完整

**问题 A：只有用户输入，没有助手回答**
- `save_user_input()` 只记录了用户说的话，助手的回复没存
- 对话的另一半丢了

**问题 B：分类是硬编码的 life**
- 不管用户聊的是工作、家人、兴趣爱好，一律存 `life`
- 分类失去了意义

**问题 C：只有原始文本，没有结构化**
- "用户叫 Allen"、"用户今天心情不好"、"用户中午吃了面条"都是 `user_input_XXXX`
- 模型看不出哪条信息更"核心"、哪条更"临时"

**解决方向**：
- 助手的回答也自动存一份（`agent_reply_XXXX` 或同一对话关联）
- 用户输入的分类可以交给模型判断（但在系统层面存），或者加一个"归类回合"
- 引入分层记忆：长期核心信息（profile/preferences）、中期事件（work/family/life 的结构化条目）、短期对话流水账

## 🟡 P1：架构层面，做了就上一个台阶

### 4. system prompt 太长且每次都完整发送

**现状**：
- `prompts/system.txt` ≈ 2000 字 ≈ 3000 tokens
- 加上 `MemoryStore.to_context_text()` 的记忆摘要
- 每次 API 调用都完整发送

**问题**：
- 输入 token 成本高，大部分是"固定开销"
- 随着记忆条目增多，记忆摘要会越来越长
- 最终会 hit context window limit

**解决方向**：
- **分层发送**：
  - 核心身份（300字以内，每次都发）—— "你是一个温暖的长期陪伴者"
  - 工具说明（第一次发，之后不发，模型记住了）
  - 记忆摘要（动态，只发和当前对话最相关的 3-5 条，不是全量 dump）
- 预计改动：`agent.py` ~40 行，`memory.py` ~20 行（加一个 `get_relevant_memory(keyword)`）

### 5. 记忆不分层，重要程度混在一起

**现状**：所有记忆条目平级存着。

**问题**：
- "用户叫 Allen"（长期核心信息）
- "用户上周项目 deadline"（中期事件）
- "用户今天中午吃了面条"（短期细节）
- 以上三条权重一样，模型不知道哪条更"重要"

**解决方向**：
- 引入分层（或者至少在存的时候标注重要程度）：
  - **长期 / core**：profile、preferences——用户本人的基本特征
  - **中期 / events**：work、family、life 里的事件和状态
  - **短期 / log**：对话流水账（`user_input_XXXX` 这一类）
- 给模型的记忆摘要里，分层展示，让模型知道"哪些信息是稳定的"

### 6. 没有上下文窗口管理

**前提**：如果上了"持续对话模式"

**问题**：对话历史 `messages` 列表会无限增长。
- token 爆掉
- 模型注意力分散（太早的对话反而干扰当前判断）

**解决方向**：
- **滑动窗口 + 总结**：
  - 保留最近 N 轮（比如最近 10 轮）
  - 更早的内容做一次"对话总结"，折叠成一条 system message
- 预计改动：`agent.py` ~40 行

### 7. temperature 调高了，但防御没跟上

**现状**：temperature 从 0.1 调到 0.8——语气更自然、更像人。

**风险**：
- 更高的"创造力"也意味着更高的"编造概率"
- 模型可能会：
  - 编造不存在的记忆条目
  - 过度解读用户情绪
  - 调用工具时参数格式出错（虽然有 try-except 兜底，但出错频率可能上升）

**解决方向**：
- 工具调用环节加强 schema validation
- 工具返回结果做二次校验（比如 `delete_memory` 真的删了吗）
- 在 system prompt 里更强调"不编造"—— 目前已经有了，但可以再强化
- 预计改动：`tools.py` ~20 行，`prompts/system.txt` 几行

## 🟠 P2：体验打磨

### 8. 输出格式粗糙——"实现细节泄露"

**现状**：用户在 console 里看到的是：
```
  💾  已自动记录用户输入到 [life] 'user_input_0001_...'（第1次）
============================================================
智能体任务:
今天被老板批评了
============================================================

--- 第 1 步: 思考中... ---
  [统计] 本轮: 1234 tokens (输入 1100, 输出 134)
...
  💡 Token 统计（共 1 步）：
     输入: 1100 tokens
     输出: 134 tokens
     总计: 1234 tokens
     估算费用: $0.0002 (约 0.00 元人民币)
```

**问题**：
- "工具调用过程"、"Token 统计"、"估算费用"——这些是系统日志，不是给用户看的
- 陪伴助手的用户界面应该**只有**：你说的话、助手说的话
- 其他一切都是"噪音"

**解决方向**：
- 区分"用户输出流"和"系统日志流"
- 默认只给用户看对话内容
- 加 `--debug` / `--verbose` 参数时才打印 Token 统计和工具调用
- 预计改动：`agent.py` ~30 行，`main.py` ~10 行

### 9. 记忆搜索是暴力字符串匹配

**现状**：`MemoryStore.search(keyword)` 遍历所有分类所有条目，做 `keyword in value`。

**问题**：
- 当前条目少（几十条），没问题
- 长期使用后（几百上千条）：效率低、不准确
- 无法语义搜索（搜"工作压力"，找不到"老板批评我"）

**解决方向**（长期）：
- 简单方案：加标签 / 关键词索引
- 高级方案：向量搜索（embedding + cosine similarity）
- 预计改动：简单方案 ~30 行，高级方案需要接入 embedding API

### 10. 交互提示不友好

**现状**：`python main.py` → "请输入你想让智能体帮你做的事情"

**问题**：
- 这个提示是"任务模式"的措辞
- 如果改成"持续对话模式"，应该说"有什么想聊的？（输入 exit 退出）"
- 没有"上一次你说到..."的衔接——每次都像第一次见面

## ⚪ P3：技术债务与未来隐患

### 11. 用 print 而不是标准 logging

**现状**：
- `print(f"  ⚠️  API 调用失败...")`
- `print(f"  💾  已自动记录...")`
- `print(f"  💡 Token 统计...")`

**问题**：
- 没有日志级别（INFO / DEBUG / WARN / ERROR）
- 不能静默模式运行
- 没有文件日志——出了问题没法回溯

**解决方向**：
- 引入 Python 标准 `logging` 模块
- 区分 console handler（用户可见）和 file handler（调试用）
- 预计改动：`agent.py` ~20 行，`memory.py` ~10 行，`models.py` ~10 行

### 12. 记忆是单文件 JSON，未来会膨胀

**现状**：所有内容存 `data/agent_memory.json`。

**问题**：
- 文件会越来越大（每次运行新增几条，一年就是几百条）
- 没有版本历史——记忆被覆盖了就找不回
- 单文件写——多进程/多线程场景有并发写风险（当前是单进程 CLI，没事，但未来扩展要注意）

**解决方向**：
- 简单方案：按月份分文件，或每次写前备份
- 高级方案：迁移到 SQLite（结构化查询、事务安全）
- 预计改动：简单方案 ~30 行，高级方案 ~100 行

### 13. MemoryStore 自增序号有理论并发风险

**现状**：
```python
seq = int(meta.get("user_input_count", 0)) + 1
meta["user_input_count"] = seq
```

**问题**：
- 如果两个进程/线程同时调用 `save_user_input`，可能读取同一个 count 然后都 +1，导致序号重复
- 当前是单进程 CLI，**实际不影响**
- 但未来如果做成服务或多实例部署，需要加文件锁或数据库自增

**解决方向**：
- 当前阶段：保持现状（文档里记一下）
- 未来：迁移到 SQLite 用 AUTOINCREMENT，或加文件锁（`fcntl`）

### 14. 缺少对话结束后的"总结回合"

**现状**：用户输入 → 模型回答 → 结束（除非模型主动调用了 `save_memory`）。

**问题**：这和 P0 #3 是同一个根源——记忆依赖模型"自觉"。系统层面自动记录了用户输入，但：
- 模型从对话中学到的"关于用户的结构化信息"（如"用户有个女儿叫小满"）没有提取出来
- 下次启动时，模型看到的是一条条 `user_input_XXXX` 流水账，不是"用户画像"

**解决方向**：
- 每次对话结束（或每 N 轮），加一个"记忆总结回合"：
  - 给模型一个任务："从刚才的对话中，提取关于用户的关键信息，用 save_memory 存到合适的分类"
  - 这一步不展示给用户，是系统后台做的
- 预计改动：`agent.py` ~30 行

### 15. 记忆工具调用太多会喧宾夺主

**现状**：记忆工具（save/recall/list/search/delete）是 8 个工具里的 6 个。

**风险**：
- 模型可能"为了存记忆而存记忆"，频繁调用记忆工具
- 用户体验：本来是来聊天的，结果系统一直在"我存一下、我查一下"

**解决方向**：
- 系统层面的自动记录（`save_user_input`）已经做了——这是基础保障
- 模型层面的"结构化记忆提取"（P3 #14 的总结回合）——这是"精加工"
- 两者结合后，可以考虑**减少暴露给模型的记忆工具数量**，或者在 system prompt 里强调"不要为了存记忆而打断对话"

## 💡 附带观察：当前架构的"技术可行性边界"

以当前的实现（纯 DeepSeek API + JSON 文件），**能做到什么程度、不能做到什么**：

| 能做到 | 需要额外工作才能做到 |
|--------|----------------------|
| 持续对话 + 历史持久化 | 真正的语义记忆搜索（需要 embedding） |
| 记忆分层 + 自动分类 | 多用户隔离（需要 user_id + 数据库） |
| 对话总结回合 | 长对话的流式总结（需要更复杂的 prompt 工程） |
| 用户输入自动记录 | 并发安全（当前是单用户 CLI） |
| Token 统计 + 费用估算 | 可视化 UI / 移动端适配 |

也就是说：**当前的技术栈足够支撑"一个温暖、长期陪伴的单用户助手"**——上面的 TODO 都是在这个基础上打磨体验和质量，不是引入新的技术依赖。
